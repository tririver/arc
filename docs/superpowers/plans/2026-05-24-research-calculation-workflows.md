# Research Calculation Workflows Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Route explicit calculation work directly through ARC research-planning, research-foundation, and research-execution workflows backed by tested `arc-llm` proposers-reviewer consensus execution.

**Architecture:** Keep `skills/arc/references/research-workflows/*.md` human-readable and thin. Put reusable retry/agreement logic in `packages/arc-llm/src/arc_llm/proposers_reviewer/consensus.py`, where it can reuse the existing proposers-reviewer runner and be unit tested. Store research artifacts as project-local JSON/Markdown files, never as hidden skill state.

**Tech Stack:** Markdown workflow references, JSON schemas/artifacts, `arc-paper` and `arc-domain` MCP/CLI, `arc-llm proposers-reviewer-*`, pytest.

## Artifact Contract

Use this project-local layout:

```text
<project-dir>/calculate/<run-id>/plan.json
<project-dir>/calculate/<run-id>/plan.md
<project-dir>/calculate/<run-id>/foundation/foundation.v001.json
<project-dir>/calculate/<run-id>/foundation/latest.json
<project-dir>/calculate/<run-id>/execute/consensus.config.json
<project-dir>/calculate/<run-id>/execute/<consensus-run-id>/
<project-dir>/calculate/<run-id>/report.md
```

`plan.json` is the workflow source of truth after planning. `foundation/*.json` is versioned because later checks may require corrected equations. `execute/` contains all proposer/reviewer artifacts generated by `arc-llm`.

- [ ] **Step 3: Write `research-plan.md`**

It must instruct agents to:
- read `context.json`, selected idea artifacts, domain Markdown/JSON, and domain summaries;

  whether any difficult step needs more substeps;

- stop and ask the user when consensus reaches the configurable recalculation limit.

- [ ] **Step 2: Implement config dataclasses**

- [ ] **Step 3: Write runner tests for all agreement cases**

Use a fake JSON runner. Cover:
- all three agree on attempt 1 -> accepted;
- two agree and reviewer marks one likely wrong -> next attempt runs only that proposer;
- two agree but reviewer cannot isolate the wrong proposer -> next attempt runs all proposers;
- all disagree -> next attempt runs all proposers;
- max recalculations reached -> status `blocked_for_user`.

- [ ] **Step 4: Implement runner**

The runner should execute each step independently:
1. Build a one-round `run_proposers_reviewer_batch` config.
2. Include locked accepted proposer outputs in `caller_context.locked_outputs`.
3. Use active proposer ids only for that attempt.
4. Read reviewer `review_payload.consensus`.
5. Choose next active proposer ids from reviewer verdict.
6. Stop with `blocked_for_user` when unresolved after the limit.

- [ ] **Step 5: Add CLI**


## Self-Review Notes

- The skill layer stays concise and command-oriented like `suggest-ideas.md`.
- The old `0_ref` workflows are used only for responsibilities: plan, foundation, execute. Their scripts, venvs, chain-status machinery, and bloated style are not reused.
- The plan respects package boundaries: paper/domain evidence comes through ARC tools; reusable multi-agent execution goes in `arc-llm`; skill files describe workflow behavior.
- The design is general across theoretical-physics domains and does not hard-code paper ids, authors, or subfield-specific keyword lists.
